线程安全之锁

Posted on 2017-06-11 | In iOS

为了提高CPU的的使用率，出现了多线程，然而多线程影响了数据的安全，为了保护多线程下的数据安全，出现了锁的概念。现在给大家介绍一下我了解的锁…

应用场景：需要锁定一个资源，阻碍当前线程。比如：多线程处理 global 的数组。

看到pthread_spinlock_t之后想到了OSSpinLock,以为内部封装了pthread_spinlock_t，在网上搜了一下源码发现OSSpinLock 内部是封装了一个叫pthread_lock_t的锁，而不是pthread_spinlock_t，我猜测pthread_lock_t的也是实现了类似pthread_spinlock_t的功能。

❤️spin lock 🆚 mutex❤️

预备知识

操作系统的线程调度：现代操作系统在管理普通线程时，通常采用时间片轮转算法(Round Robin，简称 RR)。每个线程会被分配一段时间片(quantum)，通常在 10-100 毫秒左右。当线程用完属于自己的时间片以后，就会被操作系统挂起，放入等待队列中，直到下一次被分配时间片。

原子操作：狭义上的原子操作表示一条不可打断的操作，也就是说线程在执行操作过程中，不会被操作系统挂起，而是一定会执行完。在多处理器的情况下，能够被多个处理器同时执行的操作任然算不上原子操作。因此，真正的原子操作必须由硬件提供支持，比如 x86 平台上如果在指令前面加上 “LOCK” 前缀，对应的机器码在执行时会把总线锁住，使得其他 CPU不能再执行相同操作，从而从硬件层面确保了操作的原子性。其中，test_and_set就是原子性操作。

busy-waiting & sleeping

《APUE》中原文是这样：A spin lock is like a mutex, except that instead of blocking a process by sleeping, the process is blocked by busy-waiting (spinning) until the lock can be acquired.

从实现原理上来讲:

  • Mutex属于sleep-waiting类型的锁。
  • Spin lock 属于busy-waiting类型的锁，

    举个例子

    例如在一个双核的机器上有两个线程(线程A和线程B)，它们分别运行在Core0和Core1上。
    如果线程A想要通过pthread_mutex_lock操作去得到一个临界区的锁，而此时这个锁正被线程B所持有，那么线程A就会被阻塞(blocking)，让出自己的时间片，Core0 会在此时进行上下文切换(Context Switch)将线程A置于等待队列中，此时Core0就可以运行其他的任务(例如另一个线程C)而不必进行忙等待。
    如果线程A是使用pthread_spin_lock操作去请求锁，那么线程A就会一直在 Core0上进行忙等待并不停的进行锁请求，直到得到这个锁为止(直到用完CPU分配的时间片而被系统挂起)。

总结

（1）Mutex适合对锁操作非常频繁的场景，并且具有更好的适应性。尽管相比spin lock它会花费更多的开销（主要是上下文切换），但是它能适合实际开发中复杂的应用场景，在保证一定性能的前提下提供更大的灵活度。

（2）spin lock的lock/unlock性能更好(花费更少的cpu指令)，但是它只适应用于临界区运行时间很短的场景。而在实际软件开发中，除非程序员对自己的程序的锁操作行为非常的了解，否则使用spin lock不是一个好主意(通常一个多线程程序中对锁的操作有数以万次，如果失败的锁操作(contended lock requests)过多的话就会浪费很多的时间进行空等待)。

（3）更保险（保守）的方法或许是先使用 Mutex，然后如果对性能还有进一步的需求，可以尝试使用spin lock进行调优

不再安全的OSSpinLock
OSSpinLock内部封装了pthread_lock_t而不是pthread_spinlock_t，我猜测既然命名为OSSpinLock，应该pthread_lock_t的也是实现了类似pthread_spinlock_t的功能
OSSpinLock不安全，不是说pthread_spinlock_t不安全

原因:在低优先级线程拿到锁时，高优先级线程进入忙等(busy-wait)状态，消耗大量 CPU 时间，从而导致低优先级线程拿不到 CPU 时间，也就无法完成任务并释放锁。这种问题被称为优先级反转。源于：不再安全的 OSSpinLock

解决方案：当公布OSSpinLock不安全后，大家纷纷转向了pthread_mutex:
有消息称，苹果在新系统中已经优化了 pthread_mutex 的性能，所以它看上去和 OSSpinLock 差距并没有那么大了。

facebook的KVOController在2016年5月17日时的一个Commit里将所有OSSpinLock替换成了pthread_mutex，可参看这个commit。

查看 CoreFoundation 的源码能够发现，苹果至少在 2014 年就发现了这个问题，并把 CoreFoundation 中的 spinlock 替换成了 pthread_mutex，具体变化可以查看这两个文件：CFInternal.h(855.17)、CFInternal.h(1151.16)。苹果自己发现问题后，并没有更新 OSSpinLock 的文档，也没有告知开发者，这有些让人失望。

dispatch_semaphore_t(信号量)

适用场景：比如有3台打印机，有5个线程要使用打印机，那么semaphore就会先记录好有3台，每成功被申请一次，就减1，减到0时，后面的申请就会被拒绝。

semaphore不属于锁的范畴，不过当把semaphore 的资源数量为 0 即 dispatch_semaphore_create(0)时，就可以当做mutex使用了

关于信号量，一般可以用停车来比喻

停车场剩余4个车位，那么即使同时来了四辆车也能停的下。如果此时来了五辆车，那么就有一辆需要等待。信号量的值就相当于剩余车位的数目，dispatch_semaphore_wait函数就相当于来了一辆车，dispatch_semaphore_signal就相当于走了一辆车。停车位的剩余数目在初始化的时候就已经指明了（dispatch_semaphore_create（long value）），调用一次dispatch_semaphore_signal，剩余的车位就增加一个；调用一次dispatch_semaphore_wait剩余车位就减少一个；当剩余车位为0时，再来车（即调用dispatch_semaphore_wait）就只能等待。有可能同时有几辆车等待一个停车位。有些车主没有耐心，给自己设定了一段等待时间，这段时间内等不到停车位就走了，如果等到了就开进去停车。而有些车主就像把车停在这，所以就一直等下去。

为什么需要配合pthread_mutex_lock？

因为，我们加锁的最初原因是为了保证数据的线程安全，配合mutex的目的也就是为了保证 data的安全

为什么需要 while 而不是 if ？

如果发出pthread_cond_signal，线程从pthread_cond_wait唤醒，在pthread_cond_wait未返回之前，如果有别人写一个线程去把这个data搞成NULL了，期间没有申请mutex锁。那么pthread_cond_wait应该不能被唤醒，还得继续等待，但是 if{}不会再次判断data的值，而while可以再次判断！

Objective-C中的锁

它们都实现了NSLocking协议：NSLock、NSRecursiveLock、NSCondition、NSConditionLock

NSCondition

实现原理：封装了一个互斥锁和条件变量！

NSConditionLock

实现原理：内部持有一个 NSCondition 对象，它的本质就是一个生产者-消费者模型。“条件被满足”可以理解为生产者提供了新的内容。在PINCache中的PINDiskCache使用了NSConditionLock

NSLock

NSLock 只是在内部封装了一个属性为 PTHREAD_MUTEX_ERRORCHECK的 pthread_mutex

⚠️NSLock的lock & unlock 必须在同一个线程(https://developer.apple.com/reference/foundation/nslock) trylock的返回值：YES代表成功，NO代表失败，避免与底层的函数返回混淆（pthread_mutex_t相关函数返回 0 代表成功，!0代表失败）

错误提示： * -[NSLock unlock]: lock ( ‘(null)’) unlocked from thread which did not lock it

NSRecursiveLock

NSRecursiveLock 内部封装了一个属性为 PTHREAD_MUTEX_RECURSIVE的 pthread_mutex。
递归锁允许递归调用，也就是lock多次，只是存在一个计数器累加1，等到unlock的时候，将计数器递减1。

举例说明：
在 YYThreadSafeArray 中的锁类型是资源为 1 的dispatch_semaphore_t (之前是OSSpinLock) 。在使用YYThreadSafeArray时，如果在enumerateObjectsUsingBlock里面再调用array.count的话 YYThreadSafeArray 会死锁，enumerateObjectsUsingBlock上有lock ，count也有lock了，所以造成了死锁。
如果把YYThreadSafeArray中的锁类型改成NSRecursiveLock的话，就不会造成死锁。（为什么作者没有改呢？我感觉还是因为 NSRecursiveLock执行效率低，不如 dispatch_semaphore_t）

@synchronized

这其实是一个 OC 层面的锁， 主要是通过牺牲性能换来语法上的简洁与可读。我们知道 @synchronized 后面需要紧跟一个 OC 对象，它实际上是把这个对象当做锁来使用。这是通过一个哈希表来实现的，OC 在底层使用了一个互斥锁的数组(你可以理解为锁池)，通过对对象去哈希值来得到对应的互斥锁。
